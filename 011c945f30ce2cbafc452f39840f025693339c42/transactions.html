Transactions

Transactions management in an e-commerce admin portal

Transactions

Admin portals heavily rely on tables due to their capacity to store a large amount of data. The design process of these tables, along with associated elements like page tabs and action buttons, is complex due to limited screen space and diverse user requirements.

This project showcase will focus on the design of the transaction module within the admin portal of a B2B E-commerce platform that specialises in construction-related products.

Gathering Requirements

Stakeholder Interviews

Stakeholders from both the Finance and Business Development department were interviewed.

The Finance team gave insights to existing processes and opportunies, while the Business Development team assisted in crystallizing the product vision.

Current Process

In a typical B2B direct sales

  • Each time an order is made or delivered, the order details such as order number, order amount, date, has to be logged in excel.
  • Buyers will be notified to make payment via an invoice.
  • Finance team will have to verify the payment amount

Business and Vision

A key feature of this B2B e-commerce site was to allow buyers to ‘Buy now and Pay later’, where payment can be made after the buyer receives their goods. The site serves as a middleman to receive payments from buyers, then releasing them to the supplier once the order is deemed complete.

Formulating...

With the research and interviews with stakeholders, the key objective of this project was fleshed out.

“How might we transform manual and repetitive transaction processes into an automated and more efficient one?”

Creating the user flow

With the B2B E-commerce site serving as a middleman between the Buyer and Supplier, payment stages was modified. This was then further broken down into 2 main flows to allow users to work on Buyer or Supplier transactions in a more focused manner:

  • Buyer Transaction - All payments relating to Buyer
  • Supplier Transaction - All payments relating to Supplier

Tabs vs Filters

To enable users to focus on one task at a time, the transaction stages were split into its respective sub-pages.

A dilemma in determining how these stages were to be displayed involved the choice between using tabs vs filters. As such, 2 designs were done up for A/B testing with 2 key users and 3 non-finance trained user.

Option 1

Tabs: Each stage were separated via tabs, ordered according to the payment sequence.

Option 2

Filters: Show all payment types and statuses in 1 page, but through the use of filters, allow users to segment out the payments they require.

Test Findings

Users were given some tasks that were typical of their SOP/routine.

They were then rated 0, 1, 2, where 0 is failure to complete task, 1 is completed with some difficulty, 2 is easy completion.

Option A: Tabs

3x

3 / 3 Success

Option B: Filters

2x

1 / 2 with difficulty

  • Using tabs, 3 out of 3 users completed a given task without difficulty.
  • Using filters, only 1 out of 2 users managed to complete the task with some difficult.
  • Having tabs that are laid out sequentially aided users’ understanding of the Payment flow, even if they were new to the system.

Selected option: Tabs

Filters can be kept, however, for other types of filter such as Order dates.

Multi-selection

In this payment module, users frequently perform the same actions on multiple payment entries. Providing a multi-selector in the table is expected to speed up their completion rate.

Initial Design

Each table row has its own action column, and a multi-selection check box to allow bulk actions.

However, it was noticed that 3 out of 5 users multi-selected the rows, and yet click on the action button, e.g. 'Release' payment, in one of the selected rows, thinking that it 'Releases' the selected.

Iteration 1

Bulk action button located on the bottommost pagination bar.

2 / 5 Success

3 users did not notice the button appearing and proceeded to click on the button within the action column again.

Iteration 2

Replacing Actions column with a singular button when multi-selected.

5 / 5 Success

All users noticed the change, and successfully performed the bulk action.

Iteration 3

Disabled Actions column, but with 1 main bulk action button appearing above it.

5 / 5 Success

All users also noticed the disabling, and was redirected to click on the bulk action button

Comparing between iterations 2 and 3, iteration 2 behaved vastly different from all other UI components in the design system. After discussion with the UI team, we have concluded with iteration 3.

Selected option: Disabled actions column

Search

The search bar was used to search for specific orders or transactions. However, the initial design proved unfriendly to users because it led them to click through every tab in order to find their desired item.

Initial Design

Search within each tab

The search bar was initially placed below the tabs such that the search is performed within each tab. However, it was observed that it was difficult to locate a particular order if the user does not know which status it is in.

Iteration

Overall page search

In the improved iteration, the search bar was shifted to the top, serving as an overall search for the page.

Upon search, all orders in various stages will be displayed in a dedicated search results page.

Closing thoughts

The key challenges in designing the Transactions module were to gather requirements from stakeholders, then formulating and simplifying the complex process into bite-sized tasks for users' easy comprehension and usage.

While designed as an internal tool, the lessons learnt from its development are still universally applicable. These design strategies can be revisited when designing websites or platforms that requires the use of tables for data management.
